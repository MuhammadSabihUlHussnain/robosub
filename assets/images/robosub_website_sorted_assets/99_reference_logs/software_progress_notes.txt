@everyone RoboSub - Software Progress Update 02/09/26

Good progress on software today around control architecture and wet-test behavior planning.

Confirmed that ROS2 commands publish correctly, hardware driver nodes receive reliably, and shutdown handling is clean. We refactored motordriver.py into a two-layer design:

Behavior layer (approach_controller): decides what the robot should do
Hardware layer (motordriver_hw): responsible only for how the motors / ESCs are driven

The two are connected with /motor_cmd interface which publishes simple strings (FWD, STOP, LEFT, RIGHT, UP, DOWN). This keeps perception and behavior separated from hardware. Also means we can swap motor drivers from L298Ns to ESCs without rewriting behavior logic.

New info from hardware clarified ESC control spec. Software is structured so motordriver_hw can switch from GPIO to ESC PWM with minimal changes. Software needs a basic calibration table from hardware regarding the new thruster set up. 

A full expected wet test behavior flow was designed for software:

Forward motion until detection
Stable left / right decision
Timed commit through gate
Turn around maneuver
Standby mode with depth hold, so bot does not sink and can be retasked without restarting the system.

Software architecture is in good shape (Lots of data and previous configs pulled from motordriver, thank you Zach). Aligned with ESC hardware, and ready to integrate once ESC migration is finished. 

TL;DR:

Cleaned up motor control architecture, verified ROS command flow, isolated motor issues to hardware (power / wiring), not software. Control stack is ready for ESCs; once installed, map to /motor_cmd to ESC PWM. Wet test behavior defined: move, detect, choose, commit, turn around, standby with depth hold for repeatable passes.
